Understanding SQL Queries

How Do Queries Work? The Role of SQL

To get information from the database, you need to speak its language: SQL (Structured Query Language). An SQL "query" is a command you write to ask the database to perform a task—most commonly, to retrieve specific data.

Let's break down a basic query:

SELECT FirstName, LastName FROM Employees WHERE Department = 'Sales';
  • SELECT FirstName, LastName: This tells the database which columns you want to see. Use SELECT * to see all columns.
  • FROM Employees: This specifies which table to look in.
  • WHERE Department = 'Sales': This is your filter. It sets the conditions that a row must meet to be included in the results.
  • ORDER BY Salary DESC: This optional clause sorts the results. You can sort by any column in either ascending (ASC) or descending (DESC) order.

Filtering with Operators

The WHERE clause is very powerful. You can combine multiple conditions and use various operators:

  • Comparison: =, != (or <>), >, <, >=, <=.
  • Logical: AND (both conditions must be true), OR (at least one condition must be true), NOT (inverts a condition, e.g., WHERE NOT Department = 'Sales').
  • Pattern Matching: LIKE is used for searching for patterns in text. The percent sign % is a wildcard for zero or more characters. For example, WHERE FirstName LIKE 'J%' finds all first names that start with 'J'.

Aggregate Functions

SQL can also perform calculations on your data using aggregate functions.

  • COUNT(column): Counts the number of rows. COUNT(*) is commonly used to count all rows that match the WHERE clause.
  • SUM(column): Calculates the sum of a numeric column for all rows that match the WHERE clause.

The Process in a Nutshell

  1. You write an SQL query that clearly states what information you want.
  2. The database reads and understands your command.
  3. It scans the specified table and applies your filters.
  4. It extracts columns or calculates aggregates as requested.
  5. It sorts the data if requested.
  6. Finally, it presents the finished, organized data to you as the "Query Result."

Core Database Concepts

The Structure of a Relational Database

A relational database organizes data into one or more tables. Think of it like a collection of spreadsheets that can be linked together.

  • Tables: A table is a collection of related data in a grid format. In our simulator, the Employees table holds all the information specifically about employees. A real-world application might have many tables, like Departments, Products, or Customers.
  • Fields (or Columns): Each table has columns that define the type of information being stored. In our Employees table, the columns are ID, FirstName, LastName, Department, and Salary.
  • Records (or Rows): A row represents a single, complete entry in the table. Each employee has their own row containing their specific information.

Primary Keys

A primary key is a special field in a table that uniquely identifies each record. No two records can have the same primary key value. Think of it like a student ID number or a national insurance number—it's guaranteed to be unique.

  • Purpose: Primary keys prevent duplicate data and are essential for creating relationships between different tables.
  • In our simulator: The ID column serves as the primary key for the Employees table. Each employee has a unique ID.

Data Types

Every field in a database must have a specific data type. This tells the database what kind of data to expect in that column. Choosing the correct data type is crucial for data integrity and efficiency.

  • Text (or Alphanumeric): Used for strings of text, like names or addresses (e.g., FirstName).
  • Integer: Used for whole numbers (e.g., ID).
  • Real (or Float): Used for numbers with decimal points. In our case, Salary is stored as an Integer, but it could be a Real type if it included cents.
  • Boolean: Can only hold two values, typically True or False. Useful for fields like "IsActive" or "HasPaid".
  • Date/Time: Used to store dates, times, or both. Essential for fields like "DateJoined" or "LastLogin".

Validation

Validation refers to the rules that are set on fields to ensure the data entered is sensible and correct. This is a critical step in preventing bad data from entering the database.

Examples of validation rules include:

  • Presence Check: Ensures a field is not left empty (e.g., an email address cannot be blank).
  • Format Check: Ensures data matches a specific pattern. For example, a rule could check that a mobile number starts with "0" or that an email address contains an "@" symbol.
  • Range Check: Ensures a number falls within a certain range (e.g., a rating must be between 1 and 5).
  • Length Check: Ensures text is a certain number of characters long (e.g., a password must be at least 8 characters).

While our simulator doesn't enforce these rules, they are a fundamental part of designing a real-world database.
