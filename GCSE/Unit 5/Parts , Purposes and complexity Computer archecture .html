Parts, Purposes, Complexities

An IGCSE Computing Simulation: **Computer Architecture (CPU)**

What is "Parts, Purposes, Complexities"?

This activity is a "thinking routine" developed at Harvard's Project Zero. It's designed to help you explore an object or a system (like computer architecture) in a structured way.

Purpose of the Routine

The goal is to spark curiosity and encourage you to observe carefully, looking beyond the obvious features. By breaking a system down, you can surface deeper questions and identify areas for further inquiry.

How We'll Use It

We will apply this routine to the **Central Processing Unit (CPU)**. You will move through three phases:

  • Parts: You'll identify the pieces or components of the CPU.
  • Purposes: You'll think about the purpose of the CPU as a whole, and the purpose of each individual part.
  • Complexities: You'll consider how the parts and purposes work together, and what makes the system complicated or raises questions for you.

In this simulation, the "Show Sample Answer" button for each section will remain locked until you have typed an answer in all the required boxes for that section.

Phase 1: Parts

The first prompt is: **What are its parts?**

Think about the main functional components *inside* a standard CPU (Von Neumann architecture) that you've learned about. Try to list the primary ones.

A more detailed diagram showing the CPU, its registers, and the buses connecting to Main Memory.

Hint:

Think about the part that gives orders, the part that does math, and the small, fast memory locations inside the CPU itself.

Sample Answer (Parts):

  • Control Unit (CU)
  • Arithmetic Logic Unit (ALU)
  • Registers (which include the Program Counter (PC), Accumulator (ACC), Memory Address Register (MAR), Memory Data Register (MDR), etc.)

Phase 2: Purposes

The next prompts are: **What are its purposes?** and **What are the purposes of each of these parts?**

First, what is the *overall purpose* of the CPU as a whole? Then, what is the *specific job* of each part you listed in Phase 1?

Sample Answer (Purposes):

  • Overall Purpose of CPU: To continuously **fetch, decode, and execute** instructions stored in memory to carry out tasks.
  • Purpose of Control Unit (CU): To manage and coordinate all hardware operations. It decodes instructions and sends out control signals to the correct components (like the ALU or memory).
  • Purpose of Arithmetic Logic Unit (ALU): To perform all **calculations** (e.g., addition, subtraction) and **logical operations** (e.g., AND, OR, NOT).
  • Purpose of Registers: To provide small, extremely fast, temporary storage locations *inside* the CPU for data, instructions, or addresses that are being actively used.

Phase 3: Complexities

The final prompt is: **How is it complicated?**

This is where we dig deeper. The complexity isn't just in the parts, but in how they *interact*. What makes this system work? What puzzles or questions do you have about it? Use the guiding questions below or write your own.

  • How does the Control Unit "know" whether to send a task to the ALU or to memory?
  • What is the relationship between the CPU's registers and the main RAM? How do they "talk" to each other?
  • How can a single CPU core seem to do multiple things at once (like play music and run a browser)?
  • What happens *first*? How is the whole sequence managed so it doesn't break?

The "Big Idea" (The Main Complexity):

The core complexity is **synchronization and timing**. All these parts must work together perfectly, step-by-step, at incredible speeds (billions of times per second).

This entire process is managed by a concept called the **Fetch-Decode-Execute Cycle**, which is timed by a **System Clock**. This cycle is the "engine" that answers most of these puzzles. It provides a simple, repeating sequence of steps that allows the CPU to process complex tasks one instruction at a time.

Bonus: "Taking the System Apart"

To really understand the system, let's trace the path of one instruction. This is the **Fetch-Decode-Execute Cycle**.

Imagine the CPU needs to process the instruction `ADD 100` (meaning: "add the data stored in memory address 100 to the value already in the Accumulator register").

1. FETCH

The CPU must *get* the instruction from RAM.

  • The **Program Counter (PC)** holds the address of the `ADD 100` instruction.
  • The CU copies this address to the **Memory Address Register (MAR)**.
  • The CU sends a "read" signal on the **Control Bus**.
  • The instruction is sent from RAM via the **Data Bus** to the **Memory Data Register (MDR)**.
  • The instruction is copied to the **Current Instruction Register (CIR)**.

2. DECODE

The CPU must *understand* what the instruction means.

The **Control Unit (CU)** reads the instruction in the CIR. It understands the `ADD` part (the opcode) and the `100` part (the operand). It now knows it needs to get data from memory and then use the ALU.

3. EXECUTE

The CPU must *perform* the action.

  • The CU copies the address `100` from the instruction into the **MAR**.
  • The data at address 100 is fetched (via the buses) into the **MDR**.
  • The CU sends this data and the data from the **Accumulator (ACC)** to the **ALU**.
  • The CU tells the **ALU** to perform 'ADD'.
  • The result from the ALU is written back into the **Accumulator (ACC)**.
  • The **Program Counter (PC)** is incremented to point to the next instruction.
