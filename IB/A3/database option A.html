Basic DML Syntax & Logic

-- Selecting Data
SELECT Field1, Field2
FROM TableName
WHERE Condition
ORDER BY Field1 ASC;

-- Joins (Standard IB Style)
SELECT STUDENT.Name, CLASS.Title
FROM STUDENT, CLASS
WHERE STUDENT.Class_ID = CLASS.Class_ID;

-- Aggregation
SELECT COUNT(StudentID), Class_ID
FROM ENROLLMENT
GROUP BY Class_ID
HAVING COUNT(StudentID) > 20;

1. Identify Tables: Which tables contain the data? List in FROM.

2. Link Keys: For >1 table, link PK=FK in WHERE.

3. Filter vs Aggregate: WHERE for rows, HAVING for grouped sums/counts.

Validation vs Verification A.3.4
Data Validation

Performed by the computer to ensure data is logical and allowed. Doesn't guarantee it is correct.

Examples: Range check, Type check, Presence check.

Data Verification

Performed by a human/system to ensure data matches the original source exactly.

Examples: Double-entry, Visual proofreading.

Information Systems A.1.2

A Database is simply an organized collection of data. It is only one component of an Information System.

An Information System (IS) is the broader environment. It includes:

  • The Database (Data)
  • Hardware (Servers)
  • Software (The DBMS / App)
  • Networks (Communication)
  • People (Users / DBA)
The Data Dictionary A.2.5

A critical system file that contains metadata (data about data). Without it, the DBMS cannot function.

It stores information such as:

  • Table and field names
  • Data types (e.g., Integer, String)
  • Validation rules and constraints
  • Relationships and Keys
  • Access rights for users

1. Insertion Anomaly

Scenario: You hire a new teacher for a new course, but no students are enrolled yet.
The Error: You cannot insert this course into the database because StudentID is part of the Primary Key and cannot be NULL.

2. Deletion Anomaly

Scenario: Alice drops out of MATH1. She was the only student taking it.
The Error: When you delete Alice's record to remove her from the class, you accidentally delete the fact that Ms. Ada teaches MATH1.

3. Update Anomaly

Scenario: Mr. Turing earns his PhD and needs his title updated.
The Error: His name is duplicated across multiple rows. If you update Alice's row but forget Bob's row, your database contains conflicting data.

ACID Properties A.1.4

Atomicity

Transaction is all or nothing. Uses rollback.

Consistency

Moves DB from one valid state to another.

Isolation

Transactions don't interfere (Concurrency).

Durability

Changes are permanent after commit.

Concurrency & Locking A.3.5 / A.3.6

Concurrency is the ability of multiple users to access and modify the database simultaneously.

The Problem: Lost Updates

If User A and User B both read a record and update it at the same time, one user's changes overwrite the other's.

The Solution: Locking

The DBMS applies a lock to a record being edited. Other users must wait until the lock is released.

Database Recovery A.3.7 / A.3.8

Methods used by the DBA to restore the database to a consistent state after a failure.

1
Transaction Logs

A sequential record of changes used to UNDO incomplete transactions or REDO committed ones.

2
Deferred Update

Changes are only written to the log until fully committed, preventing partial file writes.

Denormalization A.3.11

Denormalization is the deliberate introduction of redundancy into a normalized database.

Why do it?
  • Drastically speeds up read queries.
  • Reduces complex JOIN operations.
The Trade-offs
  • Reintroduces risk of update anomalies.
  • Slows down write/insert operations.

Transaction Recovery (May 2023) 4 Marks

"If the database crashes after the payment has been made but before the availability of the vehicle has been updated, explain how ACID properties ensure a consistent state."

Atomicity

The DBMS reads the Transaction Log. Seeing the transaction didn't complete, it performs a rollback of the payment, returning to the state before the booking started.

Consistency

Constraints (e.g., car stock cannot be negative) are re-verified after recovery, ensuring no illegal data states exist due to the interruption.

Data Warehouses & ETL A.4.3 / A.4.4

What is a Data Warehouse?

Unlike operational databases, a data warehouse is:

  • Subject-oriented: Focused on specific business areas (e.g., Sales).
  • Non-volatile: Historical data is read-only; past records are never altered.
  • Time-variant: Stores data as a series of snapshots over time.

The ETL Process

Importing data into the Warehouse.

Data Mining Techniques A.4.7
Cluster Analysis

Grouping records with similar characteristics without using pre-defined labels.

Classification

Assigning new records to pre-defined categories based on historical training data.

Association Rules

Discovering 'if-then' correlations. Also known as Market Basket Analysis.

Deviation Detection

Identifying outliers outside the historical norm. Prediction Use: Early warning systems for credit card fraud.

Advanced Models: Spatial Databases A.4.2

Spatial Databases

Optimized for geometric space (points, lines, polygons) using specialized R-Trees. Used in GIS and GPS.

Spatial Relationships Queries:
  • Topological: Contains, Intersects, Overlaps, Touches, Disjoint.
  • Distance (Metric): Within 5km, Nearest Neighbor.
  • Directional: North of, Left of.
Data Mining in Spatial Data:
  • Spatial Clustering: Finding geographical "hotspots" (e.g., disease outbreaks, crime zones).
  • Spatial Association: Location-based rules (e.g., "Properties near subway stations are commercial").
